Arrays

Arrays are magical beings. At first it doesn't seem all that special, a series of numbers or bits of string (or any variety of other objects), each separated by commas, nestled between two cold, square brackets. It seems a bit dull, really.

But how powerful they are! Their size shrinks and grows to convenience. The Standard Library has loads and loads of methods to use on them. And if these don't suit our needs, we may write our own methods.

The thing that fascinates me is the notion of what a method returns versus the method's side effects. This seems like an obvious distinction, but in practice it can be a little confusing.

The each method, for instance, always returns the original array it was called on. The side effects - the things that our code makes happen during iteration - are the only things that matter.

On the other hand, the map method may return an array that is different from the original array. This distinction means that, despite how similar these two methods are, it should be very obvious when we should use one or the other - do we need to return an array that is different from the original? If not, avoid map. Easy enough.

But in both cases, the original array remains the same. This is true of most methods. Calling reverse on an array will return an array that is different from the original, but the original remains the same. Sometimes, methods that end in ! will actually change the original array.

To highlight the reverse method in action:

array1 = [1, 2, 3]

array1.reverse RETURNS [3, 2, 1], but array1 is still [1, 2, 3].

array1.reverse! also returns [3, 2, 1], but now array1 is [3, 2, 1].

In this case, we could think of array1.reverse! as being equivalent to array1 = array1.reverse

This reflexive action implied by ! is not present in ALL methods that change the original array, however! This was a mistake I made early on - I saw how reverse and reverse! differed and I assumed this was a way to tell something about method behaviors just from their naming convention. Nope.

For instance, delete_if. array_1.delete_if { |x| x == 3 } will return [1, 2], but it also changes array_1 to [1, 2].
